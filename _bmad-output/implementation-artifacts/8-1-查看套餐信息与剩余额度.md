# Story 8.1: 查看套餐信息与剩余额度

Status: ready-for-review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为已登录用户，
我希望随时查看我当前的订阅套餐等级和剩余回测额度，
以便合理安排回测计划并决定是否升级。

## Acceptance Criteria

1. **Given** 用户访问个人账户页或回测界面
   **When** 调用 `GET /api/v1/users/me/quota`
   **Then** 返回当前套餐等级（`plan_level`）、本月已用次数（`used_count`）、套餐上限（`plan_limit`）、剩余次数（`remaining`）、重置时间（`reset_at`，北京时间 ISO 格式，FR3）
   **And** 响应格式为 `{"ok": true, "data": {...}}`
   **And** 未登录时返回 `401 Unauthorized`

2. **Given** 用户的 `user_quota` 记录尚不存在
   **When** 调用 `GET /api/v1/users/me/quota`
   **Then** 系统自动用 `user.plan_level` 创建默认 quota 记录并返回

3. **Given** 回测界面顶部已挂载
   **When** 页面加载完成
   **Then** 顶部显示剩余额度（`remaining / plan_limit`）和重置日期（北京时间日期部分）

4. **Given** 用户的 `remaining` 为 0（额度耗尽）
   **When** 进入回测界面
   **Then** 原"开始回测"按钮变为"升级套餐解锁更多次数"引导升级（按钮禁用回测提交，可跳转至套餐页）

    ```python
    def ensure_user_quota(user_id, plan_level="free"):
        quota = db.session.get(UserQuota, user_id)
        if quota is None:
            quota = UserQuota(user_id=user_id, plan_level=plan_level, used_count=0)
            db.session.add(quota)
            db.session.flush()
        else:
            # 同步用户最新套餐等级到 quota 记录
            if plan_level and quota.plan_level != plan_level:
                quota.plan_level = plan_level
        return quota
    ```

## Dev Notes

### 架构约束与关键模式

- **蓝图文件**：`backend/app/blueprints/users.py`，前缀 `/api/v1/users`
- **认证**：`@jwt_required()` + `get_jwt_identity()`
- **响应格式**：`ok(data)` → `{"ok": true, "data": ...}`；`error_response(code, msg, status)`
- **`format_beijing_iso`**：已在 `users.py` 导入，从 `backend/app/utils/time.py`，转换 UTC datetime → 北京时间 ISO 字符串
- **前端 API 模式**：`client.request({method, url})` — 见 `frontend/src/api/users.ts`
- **Vite/Vue 响应式**：使用 `ref`、`computed`、`onMounted`（已是 BacktestsView 现有模式）

### ⚠️ 关键陷阱 1：`quota.py` 的 plan_level 命名与架构文档不一致

**现状：** `backend/app/quota.py` 中 `PLAN_LIMITS` 使用：
```python
PLAN_LIMITS = {
    "free": 10,
    "basic": 100,
    "pro": 500,
    "enterprise": math.inf,
}
```

**架构文档定义（`architecture.md` 领域7）：** `plan_level ENUM('free','lite','pro','expert')`

**Epic 8 描述：** "4 级套餐（免费/轻量 200元/进阶 500元/专业 1000元）" → 对应 `free/lite/pro/expert`

**当前影响：** Story 8.1 只是**读取**现有数据，不改变套餐名称。`ensure_user_quota` 会用 `user.plan_level`（当前数据库中的值）。如果 DB 里的 `users.plan_level` 存的是 `'basic'`，返回的就是 `'basic'`。

**Story 8.1 的处理方式：** 维持现状，不在本 Story 修改 PLAN_LIMITS 或 plan 名称映射。将此差异记录为技术债，在 Story 8.2（发起支付/套餐激活）中统一对齐。

**Dev 不要：** 在本 Story 中重命名 `basic → lite` 或 `enterprise → expert`，这会破坏现有测试和回测额度检查逻辑。

### ⚠️ 关键陷阱 2：`remaining` 在 `plan_limit = 'unlimited'` 时的类型不一致

后端返回 `plan_limit: "unlimited"` 和 `remaining: "unlimited"`（字符串）当套餐为 `enterprise`。前端 `isQuotaExhausted` 已处理这种情况：`r !== 'unlimited' && r <= 0`。TypeScript interface 定义为 `number | 'unlimited'` — 务必遵守，不要用 `-1` 或 `Infinity` 代替。

### ⚠️ 关键陷阱 3：`ensure_user_quota` 的 `db.session.flush()` vs `commit()`

`ensure_user_quota` 内部只调用 `db.session.flush()`（写入 session 但不提交事务）。端点函数中需要显式调用 `db.session.commit()` 才能持久化新建的 quota 记录。当 quota 已存在时，commit 对已有数据无害。

### ⚠️ 关键陷阱 4：BacktestsView 的现有提交按钮结构

读取 `BacktestsView.vue` 确认现有"开始回测"按钮的精确 selector/class，再做替换。**不要** 猜测 class 名称或按钮位置——先 Read 文件再修改。

### ⚠️ 关键陷阱 5：`reset_at` 可能为 `None`

`UserQuota.reset_at` 字段为 `nullable=True`。新建 quota 记录时未设置 `reset_at`（由 Story 8.4 的 Celery Beat 月度重置任务来维护）。前端模板使用 `v-if="quota.reset_at"` 做条件渲染，后端 `format_beijing_iso(None)` 返回 `None`，前端 interface 定义为 `string | null`。
