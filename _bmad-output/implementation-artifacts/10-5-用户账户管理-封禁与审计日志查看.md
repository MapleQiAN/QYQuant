# Story 10.5: 用户账户管理（封禁与审计日志查看）

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为管理员，
我希望查看用户列表、封禁违规用户，并查看审计日志，
以便处理违规行为并追溯问题。

## Acceptance Criteria

1. **Given** 管理员访问用户管理页面
   **When** 调用 `GET /api/v1/admin/users`（支持按手机号/昵称搜索）
   **Then** 返回用户列表（昵称、注册时间、套餐、封禁状态，FR54）

2. **Given** 管理员点击"封禁用户"
   **When** 调用 `PATCH /api/v1/admin/users/:id`（is_banned=true）
   **Then** users.is_banned=true，该用户所有 Refresh Token 写入 Redis 黑名单（强制登出）
   **And** 被封禁用户登录时返回 403 并提示"账号已被封禁"
   **And** 操作写入 audit_logs（action="user_ban"，NFR29）

3. **Given** 管理员查看审计日志
   **When** 调用 `GET /api/v1/admin/audit-logs`（支持按 operator_id/action/target 筛选）
   **Then** 返回审计日志列表，覆盖支付操作、管理员操作、策略代码访问记录（NFR29）
   **And** 审计日志只读，无任何修改或删除接口（不可篡改，NFR29）
   **And** 日志保留期 ≥90 天，超期记录归档而非删除

## Dev Notes

### 关键架构约束

- **Admin Blueprint**: 所有端点在 `backend/app/blueprints/admin.py`，使用 `@require_admin` 装饰器
- **响应格式**: `ok(data, meta={total, page, per_page})` 或 `error_response(code, message, status_code)`
- **时间处理**: 数据库存 UTC (`now_utc()`)，API 输出北京时间 (`format_beijing_iso()`)
- **审计日志**: 使用 `log_audit(operator_id, action, target_type, target_id, details)` + `db.session.commit()`
- **通知集成**: `create_notification(user_id, type, title, content)` from `app.services.notifications`
- **异步邮件**: `send_email_notification.delay(user_id, event_type, context_data)`

### Redis Token 黑名单实现要点

- 架构要求 (ARCH-5): Redis 用不同 DB number 存 Token 黑名单
- 封禁时需将用户所有活跃 Refresh Token 加入黑名单
- 参考现有 auth 蓝图中的 Token 刷新/登出逻辑来确定 Redis key 格式
- 黑名单条目设置 TTL 等于 Token 剩余有效期（避免永久占用 Redis 内存）

### 手机号脱敏规则

- 用户列表中手机号显示为脱敏格式：`138****1234`（中间4位用*替换）
- 后端处理脱敏，前端直接展示

### 审计日志不可篡改性保障

- **仅暴露 GET 端点**：不提供任何 PUT/PATCH/DELETE 审计日志的 API
- **保留期 ≥90 天**：超期记录归档而非物理删除（本故事可先不实现归档逻辑，但设计上预留）
- **现有审计动作一览**：strategy_approve, strategy_reject, strategy_takedown, report_dismiss, job_terminate
- **本故事新增审计动作**：user_ban, user_unban
