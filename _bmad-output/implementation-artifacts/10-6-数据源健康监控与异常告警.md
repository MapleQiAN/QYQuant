# Story 10.6: 数据源健康监控与异常告警

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为管理员，
我希望平台自动监控聚宽 JQData API 健康状态，异常时立即通知，
以便及时处置数据源故障，减少对回测服务的影响。

## Acceptance Criteria

1. **Given** Celery Beat 每 5 分钟触发一次数据源健康检查
   **When** 执行 JQData 健康探测任务
   **Then** 复用现有 `JoinQuantClient` 完成一次轻量探测（受现有超时/重试配置约束）
   **And** 持久化当前状态（`healthy` / `unhealthy` / `unknown`）、`last_checked_at`、`last_success_at`、`last_failure_at`、`last_error_message`、连续失败次数

2. **Given** JQData API 超时、认证失败或返回错误
   **When** 数据源状态从 `healthy` 或 `unknown` 变为 `unhealthy`
   **Then** 向所有配置了邮箱的 admin 用户异步发送告警邮件"数据源异常：JQData API 不可用"（FR56）
   **And** `GET /api/v1/admin/data-source-health` 返回状态为"异常"、颜色标识为红色、并包含最近错误信息与最后检查时间
   **And** 在持续故障期间不重复轰炸邮件（仅首次进入异常态发送一次）

3. **Given** 数据源此前处于 `unhealthy`
   **When** 后续健康检查恢复成功
   **Then** 状态自动更新为"正常"
   **And** 向所有配置了邮箱的 admin 用户发送一次恢复通知邮件
   **And** 管理后台健康看板在下次查询时展示绿色"正常"状态，无需人工修复数据

## Dev Notes

### 关键架构约束

- **定时调度必须走现有 Celery 配置**: `backend/app/celery_app.py` 已通过 `celery_app.conf.beat_schedule` 管理定时任务；10.6 直接沿用该模式，不引入额外调度器
- **JQData 探测复用现有客户端**: `backend/app/providers/joinquant.py` 已封装账号鉴权、请求超时和重试；默认超时来自 `JQDATA_REQUEST_TIMEOUT_SECONDS`（3 秒），默认重试来自 `JQDATA_MAX_RETRIES`（2 次）
- **Admin API 继续扩展 `admin.py`**: 所有后台接口统一放在 `backend/app/blueprints/admin.py`，使用 `@require_admin`、`ok()`、`error_response()` 这套现有模式
- **邮件必须异步发送**: 告警与恢复通知走 `send_email_notification.delay(...)`，不要在健康检查任务里直接调用 Flask-Mail
- **状态展示以持久化结果为准**: 前端看板只读状态表/API 结果，不直接触发真实探测，避免页面刷新引发额外外部调用

### 已存在的基础设施（勿重复创建）

- **JoinQuantClient** (`backend/app/providers/joinquant.py`): 已实现鉴权、超时控制、错误包装，异常统一抛 `JoinQuantAPIError`
- **MarketDataService** (`backend/app/services/market_data.py`): 已封装缓存查询与 provider 拉取逻辑，但健康检查不要直接复用 `get_market_data()` 作为探测入口，以免把监控流量和行情缓存写入耦合在一起
- **邮件通知任务** (`backend/app/tasks/notification_tasks.py`): 已具备异步发送、最多 3 次重试、幂等保护能力
- **邮件模板工具** (`backend/app/utils/email_templates.py`): 已有策略审核/下架模板，10.6 直接在同一文件扩展
- **Admin 前端骨架** (`frontend/src/api/admin.ts`, `frontend/src/stores/useAdminStore.ts`, `frontend/src/views/admin/AdminDashboard.vue`): 已建立 API + Store + 页面联动模式

### 监控设计护栏

- **不要新建裸 `requests` 探针**: 外部数据源健康检查必须复用 `JoinQuantClient`，这样才能继承现有账号配置、超时和错误语义
- **不要把异常检查写进 `/admin/data-source-health` 接口**: 接口只负责读状态；真实探测仅由 Celery Beat 执行
- **不要复用 `audit_logs` 记录周期性健康波动**: `audit_logs` 主要用于管理员关键操作；健康波动应写入专用状态表和应用日志
- **不要持续重复发邮件**: 仅在状态迁移时发送邮件，连续失败期间只更新 `last_checked_at`、`last_error_message`、`consecutive_failures`
- **首次部署状态不能默认显示正常**: 如果从未执行过探测，前端必须显示 `unknown / 未检测`

### 前端模式参考

- **后台入口卡片模式**: 参考 `AdminDashboard.vue` 现有的策略审核、用户管理、任务监控卡片样式
