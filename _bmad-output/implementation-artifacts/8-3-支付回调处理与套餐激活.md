# Story 8.3: 支付回调处理与套餐激活

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为平台，
我希望安全处理微信/支付宝的支付回调并立即激活用户的新套餐，
以便用户付款后无感知地获得升级权益。

## Acceptance Criteria

1. **Given** 用户完成支付，支付平台向 Webhook 端点发送回调
   **When** `POST /api/v1/payments/webhook/wechat`（或 `/alipay`）收到有效回调
   **Then** 验证回调签名（沙箱模式跳过），将 `payment_orders.status` 更新为 `paid`
   **And** 在 `subscriptions` 表创建记录（`plan_level`、`starts_at=now`、`ends_at=now+1个月`）
   **And** 更新 `users.plan_level` 为新套餐等级
   **And** 更新 `user_quota.plan_level` 为新套餐等级（调用 `ensure_user_quota`）
   **And** 发送站内通知"套餐升级成功，已解锁 X 次/月回测额度"（FR59，expert 套餐显示"无限次"）
   **And** 将支付激活操作写入 `audit_logs`（`action="subscription_activated"`，NFR29）
   **And** 返回 `{"ok": true, "data": {"message": "ok"}}` HTTP 200

2. **Given** 支付平台重复发送同一回调（幂等场景）
   **When** `POST /api/v1/payments/webhook/wechat`（或 `/alipay`）再次收到已处理的回调
   **Then** 检测到 `payment_orders.status` 已为 `paid`
   **And** 跳过激活流程，直接返回 HTTP 200（不重复创建 subscription、notification、audit_log）

3. **Given** 收到无效签名的回调（非沙箱模式）
   **When** 签名校验失败
   **Then** 返回 HTTP 400，不执行任何数据库写入

4. **Given** 回调中携带不存在的 `order_id`
   **When** 在 `payment_orders` 表中查不到对应记录
   **Then** 返回 HTTP 400（安全：不泄露内部信息，统一返回 `INVALID_CALLBACK`）

    ```python
    def _activate_subscription(order, provider_order_id=None):
        """
        原子性地激活订阅。
        - 更新 PaymentOrder.status = 'paid'
        - 创建 Subscription 记录（1个月有效期）
        - 更新 User.plan_level 和 UserQuota.plan_level
        - 写入 AuditLog
        - 发送站内通知
        注意：调用者需在函数外部负责 db.session.commit()
        """
        starts_at = now_utc()
        ends_at = starts_at + timedelta(days=31)

        # 更新订单状态
        order.status = 'paid'
        if provider_order_id:
            order.provider_order_id = provider_order_id

        # 创建订阅记录
        subscription = Subscription(
            user_id=order.user_id,
            plan_level=order.plan_level,
            starts_at=starts_at,
            ends_at=ends_at,
            status='active',
            payment_provider=order.provider,
            payment_order_id=order.id,
        )
        db.session.add(subscription)

        # 更新用户套餐等级
        user = db.session.get(User, order.user_id)
        if user:
            user.plan_level = order.plan_level

        # 更新配额套餐上限
        ensure_user_quota(order.user_id, order.plan_level)

        # 写入审计日志
        audit = AuditLog(
            operator_id=None,  # 系统操作，非用户主动操作
            action='subscription_activated',
            target_type='user',
            target_id=order.user_id,
            details={
                'order_id': order.id,
                'plan_level': order.plan_level,
                'provider': order.provider,
                'amount': float(order.amount),
            },
        )
        db.session.add(audit)

        # 发送站内通知
        quota_display = serialize_plan_limit(order.plan_level)
        if quota_display == 'unlimited':
            quota_text = '无限次'
        else:
            quota_text = f'{quota_display} 次'
        create_notification(
            user_id=order.user_id,
            type='subscription_activated',
            title='套餐升级成功',
            content=f'您已成功升级为 {order.plan_level} 套餐，已解锁 {quota_text}/月回测额度。',
        )
    ```

    ```python
        """
        微信支付回调端点。
        - 不需要 JWT 认证（支付平台调用，非用户）
        - 沙箱模式：跳过签名验证，从请求体读取 order_id

        """

        else:

            return error_response('INVALID_CALLBACK', '无效回调参数', 400)

        order = db.session.get(PaymentOrder, order_id)
        if order is None:
            return error_response('INVALID_CALLBACK', '无效回调参数', 400)

        # 幂等处理
        if order.status == 'paid':
            return ok({'message': 'ok'})

        _activate_subscription(order, provider_order_id=provider_order_id)
        db.session.commit()

        return ok({'message': 'ok'})
    ```

    ```python
            return 'fail', 400

        order = db.session.get(PaymentOrder, order_id)
        if order is None:
            return 'fail', 400

        # 幂等处理
        if order.status == 'paid':
            return 'success', 200

        _activate_subscription(order, provider_order_id=provider_order_id)
        db.session.commit()
    ```

## Dev Notes

### 架构约束与关键模式

- **Webhook 无 JWT 认证**：`/webhook/wechat` 和 `/webhook/alipay` 端点由支付平台调用，不使用 `@jwt_required()`。安全依赖签名验证（沙箱模式下跳过）。
- **响应格式差异**：微信 Webhook → `ok({'message': 'ok'})` JSON；支付宝 Webhook → 纯文本 `"success"` 或 `"fail"`（Flask 返回元组 `('success', 200)`）。
- **原子性事务**：所有激活操作（PaymentOrder、Subscription、User、UserQuota、AuditLog、Notification）在同一个 `db.session` 中，函数最后 `db.session.commit()`。
- **`ensure_user_quota` 用法**：传入 `user_id` 和 `plan_level` 会自动创建或更新 `user_quota.plan_level`。已在 `quota.py` 实现，直接调用即可。
- **通知服务**：`create_notification(user_id, type, title, content)` 在 `backend/app/services/notifications.py`。**不** 提交事务，由调用者 commit。
- **审计日志**：`operator_id=None` 表示系统自动操作（非用户主动）。`target_type='user'`，`target_id=user_id`。

### ⚠️ 关键陷阱 1：端点不在 BluePrint swagger 规范下

`payments.py` 使用 `flask_smorest.Blueprint`。`/webhook/*` 端点返回格式特殊（支付宝返回纯文本），如果使用 smorest 的 schema 装饰器会有问题。**不要**在 Webhook 端点上加 `@bp.response(...)` schema 装饰器，直接 `return` Flask response 即可。

### ⚠️ 关键陷阱 2：subscription 有效期计算

Story 规划中套餐按月计费，但 Story 8.4 才实现月度额度重置。本 Story 中 `ends_at = starts_at + timedelta(days=31)` 是简化处理（不精确到月末）。Story 8.4 实现后，Celery Beat 会在月初重置 `used_count`，`ends_at` 仅用于标记订阅状态。

### ⚠️ 关键陷阱 3：支付宝 form-data 格式

真实支付宝通知使用 `application/x-www-form-urlencoded` POST，字段为 `out_trade_no`（对应 `order.id`）、`trade_no`（支付宝流水号）等。沙箱测试时允许 JSON 格式，但测试代码中使用 JSON body 而非 form-data。

### ⚠️ 关键陷阱 4：幂等检查必须在事务开始前

幂等检查 `if order.status == 'paid': return ...` 必须在任何数据库写入之前执行，且不需要加锁（并发重复回调的极端情况由数据库唯一约束兜底，本 Story 沙箱阶段不需要悲观锁）。

### ⚠️ 关键陷阱 5：通知文本中的额度显示

`serialize_plan_limit('expert')` 返回字符串 `'unlimited'`，需要转换为"无限次"。其他套餐返回整数（如 `200`），格式化为 `"200 次"`。不要直接显示 `"unlimited"` 给用户。

### ⚠️ 关键陷阱 6：测试中 `_login_user` 使用固定验证码

测试 fixture 依赖 `AUTH_FIXED_SMS_CODE = '123456'`（在 `conftest.py` 的 `app` fixture 中通过 `create_app({'TESTING': True, ...})` 传入，或由 `TestConfig` 提供）。先确认 `conftest.py` 中的 app fixture 包含此配置，否则 `_seed_code` 无效。

### ⚠️ 关键陷阱 7：`_activate_subscription` 不调用 `db.session.commit()`

`_activate_subscription` 是辅助函数，负责添加所有对象到 session 但**不 commit**。commit 由调用方（Webhook endpoint）负责。这样保证原子性：任何步骤失败，整个事务回滚。
