# Story 10.7: 注销用户的已发布策略处理

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为平台，
我希望当策略发布者注销账号时，正确处理其已发布的策略，
以便保障策略广场内容完整性和其他用户的使用体验。

## Acceptance Criteria

1. **Given** 用户注销账号（调用 `DELETE /api/v1/users/me`，Story 2.4 触发）
   **When** 系统处理注销后置逻辑
   **Then** 该用户在广场公开的策略（`is_public=True`）保持可见，作者显示为"已注销用户"（FR65）

2. **Given** 用户注销账号
   **When** 系统处理注销后置逻辑
   **Then** 该用户私有策略（`is_public=False`）标记 `deleted_at`，不再出现在该用户的策略列表中

3. **Given** 其他用户（User B）已导入被注销用户（User A）的策略
   **When** User A 注销账号
   **Then** User B 的导入记录（`owner_id=B.id, source_strategy_id=A的策略id`）保持可用，不受影响

4. **Given** 被注销用户发布的帖子和评论
   **When** 用户注销后其他用户浏览帖子/评论
   **Then** 作者昵称显示为"已注销用户"（由 Story 2.4 已将 `users.nickname` 改为"已注销用户"自动实现）

## Dev Notes

### 关键架构约束

- **注销逻辑入口唯一**：所有账号注销后置处理必须集中在 `backend/app/blueprints/users.py` 的 `DELETE /api/v1/users/me` 端点，与 Story 2.4 已有的用户软删除逻辑合并，在同一 `db.session.commit()` 前完成，保证事务原子性
- **公开策略零改动**：`is_public=True` 的策略记录不添加 `deleted_at`，策略广场(`_marketplace_visibility_clause()`)已有 `is_public=True AND review_status='approved'` 过滤，保持可见；作者名称通过 `users.nickname`（已被改为"已注销用户"）JOIN 展示，无需修改策略数据
- **帖子/评论零改动**：`Post.user_id` 和 `Comment.user_id` 保留指向已注销用户的外键；前端/接口通过 JOIN `users.nickname` 展示"已注销用户"，Story 2.4 已处理 nickname 变更
- **不触碰他人导入记录**：`UPDATE` 语句必须限定 `owner_id = user.id`，User B 的导入副本（`owner_id = B.id`）不受影响

### 已存在的基础设施（勿重复创建）

- **`DELETE /api/v1/users/me` 端点**（`backend/app/blueprints/users.py` 第 95-123 行）：Story 2.4 已实现软删除用户、吊销 Token、写审计日志；本故事在 `db.session.commit()` 前插入策略软删除逻辑即可
- **`now_utc()` 工具函数**（`backend/app/utils/time.py`）：已在 users.py 中导入使用，用于一致的时间戳生成
- **`Strategy` 模型**（`backend/app/models.py` 第 101-167 行）：已有 `owner_id`、`is_public`、`deleted_at`（**需新增**）字段
- **`_marketplace_visibility_clause()`**（`backend/app/blueprints/marketplace.py` 第 626 行）：已有 `is_public=True AND review_status='approved'` 过滤，软删除的私有策略不会进入此路径
- **`list_strategies()` 端点**（`backend/app/blueprints/strategies.py` 第 139-161 行）：目前只过滤 `owner_id`，需追加 `deleted_at IS NULL` 条件

**注销端点插入逻辑位置（users.py 第 106-114 行间）：**

    Strategy.deleted_at.is_(None),

### 关于帖子/评论展示

Story 2.4 已将注销用户 `nickname` 设为"已注销用户"、`avatar_url` 清空。帖子和评论通过 `user_id` FK 关联，查询时 JOIN `users` 表取 `nickname`，自动展示"已注销用户"。本故事**无需修改** Post/Comment/PostComment/Comment 相关代码。

### 迁移注意事项

- 新增 `deleted_at` 列默认 `nullable=True`，不影响已有记录
- 生成迁移后检查 `op.add_column('strategies', sa.Column('deleted_at', sa.DateTime(timezone=True), nullable=True))`
- 如需在 PostgreSQL 上快速建立过滤索引（可选）：`CREATE INDEX CONCURRENTLY ix_strategies_not_deleted ON strategies (owner_id) WHERE deleted_at IS NULL;`

### 近期实现模式与可复用经验

- **Story 2.4 已确立软删除模式**：`users.deleted_at`、`SimulationBot.deleted_at`（第 305 行）均采用相同 `db.DateTime(timezone=True), nullable=True` 模式
- **批量 UPDATE 使用 `synchronize_session=False`**：避免 SQLAlchemy ORM 会话同步开销，适合此类后置清理场景
- **事务原子性**：所有后置操作（用户脱敏 + Token 吊销 + 策略软删除 + 审计日志）必须在同一 `db.session.commit()` 内完成
