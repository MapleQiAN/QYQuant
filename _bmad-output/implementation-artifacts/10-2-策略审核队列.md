# Story 10.2: 策略审核队列

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为管理员，
我希望查看待审核的策略列表，并可通过或拒绝策略发布申请，
以便管控平台内容质量。

## Acceptance Criteria

1. `GET /api/v1/admin/strategies?review_status=pending&page=1&per_page=20` 仅 admin 可访问，返回待审核策略分页列表，字段至少包含 `id`、`title/name`、`description`、`category`、`tags`、`display_metrics`、`owner_id`、`author_nickname`、`created_at`、`review_status`，并按提交时间倒序排序。
2. `PATCH /api/v1/admin/strategies/:id/review` 仅 admin 可访问；请求体 `status` 仅允许 `approved` 或 `rejected`；当 `status=rejected` 时 `reason` 必填；不存在的策略返回 404，非 `pending` 状态的重复审核返回 409。
3. 审核通过时更新 `strategies.review_status='approved'` 且 `is_public=true`；审核拒绝时更新 `strategies.review_status='rejected'` 且 `is_public=false`；两种情况均同步刷新 `updated_at` 和 `last_update`，保证与 marketplace 可见性规则一致。
4. 审核结果会通知策略作者：必须写入站内通知；同时复用现有异步邮件任务发送审核结果邮件，邮件内容包含策略标题、审核结果，拒绝时包含拒绝原因。
5. 审核操作必须写入 `audit_logs`；审核通过使用 `action="strategy_approve"`，审核拒绝使用 `action="strategy_reject"`；`details` 至少包含 `review_status_before`、`review_status_after`、`reason`、`strategy_owner_id`。
6. 前端提供管理员审核页，支持查看 pending 列表、点击通过、点击拒绝并填写原因、操作成功后局部刷新列表并给出 toast 反馈；非 admin 用户继续由现有路由守卫拦截。

## Dev Notes

### 架构约束与复用点

**Admin 骨架已存在，不要重复造轮子**
- `backend/app/blueprints/admin.py` 已注册 `/api/v1/admin/health`，本 Story 应直接扩展同一 Blueprint。 [Source: backend/app/blueprints/admin.py]
- `backend/app/utils/auth_helpers.py` 已提供 `@require_admin`，这是当前项目唯一的 admin 权限门禁。 [Source: backend/app/utils/auth_helpers.py]
- 前端已存在 `/admin` 路由、`AdminDashboard.vue`、`api/admin.ts`、`useAdminStore.ts`，策略审核页应在此基础上延展。 [Source: frontend/src/router/index.ts] [Source: frontend/src/views/admin/AdminDashboard.vue] [Source: frontend/src/api/admin.ts] [Source: frontend/src/stores/useAdminStore.ts]

**策略发布与审核数据链路已存在一半**
- `Strategy` 模型已经有 `is_public`、`review_status`、`display_metrics`、`owner_id` 等审核所需字段，无需新增审核表。 [Source: backend/app/models.py]
- 用户提交策略到 marketplace 时，`POST /api/v1/marketplace/strategies` 已把策略状态改成 `pending`，并写入提交通知与审计日志；本 Story 的审核队列直接消费这批 `pending` 数据。 [Source: backend/app/blueprints/marketplace.py]
- marketplace 对外展示策略的可见性条件是 `is_public=True` 且 `review_status='approved'`，因此审核通过必须同时写这两个字段，拒绝时必须确保 `is_public=False`。 [Source: backend/app/blueprints/marketplace.py]

**通知与审计能力优先复用**
- 站内通知复用 `backend/app/services/notifications.py:create_notification`，其底层通过 `Notification` 模型写入未读记录。 [Source: backend/app/services/notifications.py] [Source: backend/app/utils/notification.py]
- 邮件通知复用现有 Celery 任务 `backend/app/tasks/notification_tasks.py:send_email_notification`，不要在 admin 路由里直接调用 `mail.send()`。 [Source: backend/app/tasks/notification_tasks.py] [Source: backend/app/celery_app.py]
- 审计日志复用 `backend/app/utils/audit.py:log_audit` / `write_audit_log`，保持所有管理操作记录格式一致。 [Source: backend/app/utils/audit.py]

**时间与返回格式约束**
- `Strategy.created_at` / `updated_at` / `last_update` 当前是毫秒时间戳，序列化时应复用 `format_beijing_iso_ms()`，不要按 `datetime` 直接处理。 [Source: backend/app/models.py] [Source: backend/app/utils/time.py]
- `Notification.created_at` 是带时区时间，通知列表已有北京时间输出约定，审核结果文案与时间格式应与现有通知系统保持一致。 [Source: backend/app/models.py] [Source: backend/tests/test_notifications.py]
- API 响应统一走 `ok()` / `error_response()`，延续当前 `/api/v1/admin/*` 与 marketplace 端点的错误格式。 [Source: backend/app/blueprints/admin.py] [Source: backend/app/blueprints/marketplace.py]

### 实现边界与注意事项

- 本 Story 只处理“策略审核队列”本身，不引入新的审核工作流引擎、不新增审核表、不做批量审核。
- 审核接口只处理 `pending` 项，已经 `approved` 或 `rejected` 的记录不允许再次 PATCH 覆盖，避免内容运营误操作。
- 列表页优先展示审核决策所需摘要，不要暴露 `code_encrypted`、`code_hash`、原始策略文件路径等敏感信息。
- 拒绝理由属于审核结果的一部分，需要同时进入作者通知与审计日志；如果未来需要让作者在前台看到理由，本 Story 的后端返回字段应为此预留干净的数据结构。

### Project Structure Notes

- 后端继续沿用 `backend/app/blueprints/admin.py` 承载 admin 接口；不要平行再建 `admin_strategies.py`，除非当前 Blueprint 已明显过大。
- 前端页面放在 `frontend/src/views/admin/StrategyReview.vue`，与现有 `AdminDashboard.vue` 同目录。
- admin API 客户端继续放在 `frontend/src/api/admin.ts`；若 store 拆分，优先放在 `frontend/src/stores/` 下保持 Pinia 结构一致。
