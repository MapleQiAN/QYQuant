# Story 10.4: 回测任务队列监控与手动干预

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为管理员，
我希望查看回测任务队列状态，并能手动终止卡死的任务，
以便保障回测服务的稳定运行。

## Acceptance Criteria

1. **队列统计信息：** `GET /api/v1/admin/backtest/queue-stats` 仅 admin 可访问；返回当前排队数（pending count）、运行中数（running count）、平均耗时（基于近 100 条已完成任务）、近 1 小时失败率（failed+timeout / total in last hour）（FR52）。
2. **疑似卡死任务列表：** 同一接口返回所有 `status=running` 且 `started_at` 距今超过 10 分钟的任务列表（含 job_id、user_id、strategy_id、策略名称、started_at、运行时长），按运行时长倒序（FR52）。
3. **手动终止任务：** `DELETE /api/v1/admin/backtest/:job_id` 仅 admin 可访问；调用 `celery_app.control.revoke(job_id, terminate=True)` 撤销 Celery 任务；更新 `backtest_jobs.status=failed`、`error_message="管理员手动终止"`、`completed_at=now_utc()`（FR53）。
4. **审计日志：** 终止操作写入 `audit_logs`（action="job_terminate"，target_type="backtest_job"，details 含 job_id、user_id、strategy_id、running_duration、admin_note）（NFR29）。
5. **终止后通知：** 终止操作触发站内通知给任务所属用户（type="job_terminated"，内容含策略名称和终止原因）；不发送邮件通知（非关键告警）。
6. **前端实现：** 管理员后台新增任务监控页面 `/admin/backtest-monitor`；展示队列统计卡片（排队数、运行中、平均耗时、失败率）；展示疑似卡死任务表格，支持"终止任务"操作按钮；终止时可输入备注（可选）；操作后局部刷新并给出 toast 反馈；页面支持手动刷新按钮。

### 实现边界与注意事项

- **不需新建任何数据模型**，BacktestJob 和 AuditLog 已存在，直接查询即可。
- **"卡死"判定标准**：`status=running` 且 `started_at` 距今超过 10 分钟（超过 Celery soft_time_limit 的 2 倍）。
- **终止操作幂等性**：已完成/已失败的任务返回 409，不重复操作。
- **Celery revoke 的边界**：`revoke(terminate=True)` 向 worker 发送 SIGTERM，但如果 worker 已经完成任务（DB 状态延迟更新），revoke 是无害的空操作。
- **pending 任务终止**：pending 任务尚未被 worker 拉取，revoke 可以取消其执行；DB 状态同样更新为 failed。
- **统计接口不分页**：卡死任务通常极少（0~5 个），无需分页；如果异常大量卡死，前端可截取前 50 条展示。
- **前端不需自动轮询**：提供手动刷新按钮即可，避免频繁请求后端。
- **user_id 可能为 null**：BacktestJob.user_id 允许 nullable，终止通知前需检查 user_id 是否存在。
