# Story 7.5: 停止与删除模拟机器人

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为已登录用户，
我希望能暂停或彻底删除模拟托管机器人，
以便管理我的模拟托管槽位。

## Acceptance Criteria

1. **Given** 用户点击"暂停机器人"
   **When** 调用 `PATCH /api/v1/simulation/bots/:id`（body: `{"status": "paused"}`）
   **Then** 机器人状态更新为 `paused`，Celery Beat 定时任务跳过该机器人（FR47）
   **And** 返回更新后的机器人数据（含 id、status、updated_at）

2. **Given** 机器人当前为 `paused` 状态
   **When** 调用 `PATCH /api/v1/simulation/bots/:id`（body: `{"status": "active"}`）
   **Then** 机器人状态恢复为 `active`，恢复定时执行
   **And** 返回更新后的机器人数据

3. **Given** 用户点击"删除机器人"并确认
   **When** 调用 `DELETE /api/v1/simulation/bots/:id`
   **Then** `simulation_bots` 记录软删除（`deleted_at` 写入当前 UTC 时间），释放一个槽位
   **And** 相关历史记录（`simulation_records`、`simulation_positions`）保留不删除
   **And** 返回 `{"deleted": true}`

4. **Given** 机器人被软删除后
   **When** 调用 `GET /api/v1/simulation/bots`
   **Then** 已删除机器人不出现在列表中

5. **Given** 机器人被软删除后
   **When** Celery Beat 定时任务运行
   **Then** 已删除机器人不被执行

6. **Given** 用户尝试操作不属于自己的机器人或已删除的机器人
   **When** 调用 `PATCH` 或 `DELETE /api/v1/simulation/bots/:id`
   **Then** 返回 `404` 和错误码 `BOT_NOT_FOUND`

7. **Given** 用户未登录
   **When** 调用 `PATCH` 或 `DELETE /api/v1/simulation/bots/:id`
   **Then** 返回 `401 Unauthorized`

8. **Given** PATCH 请求 body 中 status 不是 `"paused"` 或 `"active"`
   **When** 调用 `PATCH /api/v1/simulation/bots/:id`
   **Then** 返回 `422` 和错误码 `VALIDATION_ERROR`

## Dev Notes

### 架构约束与关键模式

- **蓝图文件**：`backend/app/blueprints/simulation.py`（所有 Stories 7.x 端点在此文件）
- **HTTP 方法装饰器**：`flask-smorest` Blueprint 支持 `@bp.patch(...)` 和 `@bp.delete(...)` —— 与已有 `@bp.get` / `@bp.post` 风格一致
- **认证**：所有端点使用 `@jwt_required()` + `get_jwt_identity()`
- **响应格式**：`ok(data)` 返回 `{"data": ..., "ok": true}`，`error_response(code, msg, status)` 返回错误结构
- **软删除模式**：设置 `deleted_at = now_utc()`；所有查询须加 `deleted_at.is_(None)` 过滤
- **`now_utc`**：位于 `backend/app/utils/time.py`，已在 `simulation_tasks.py` 中导入，在 `simulation.py` 蓝图中需显式添加导入
- **前端 API 模式**：`client.request({method, url, data})` —— 见 `frontend/src/api/simulation.ts`
- **Pinia Store 乐观更新**：PATCH 成功后直接修改 `bots` 数组中对应项的 `status`，无需重新 fetch 整个列表
- **删除后 store 同步**：`deleteBot` 直接过滤 `bots` 数组，无需 refetch

### ⚠️ 关键陷阱 1：`SimulationBot.deleted_at` 是新增字段，需迁移

`simulation_bots` 表目前**不包含** `deleted_at` 列。必须先完成 Task 1 + Task 2（模型更新 + Alembic 迁移），再运行任何后端测试或端点调用，否则所有相关查询报 `sqlalchemy.exc.ProgrammingError: column simulation_bots.deleted_at does not exist`。

```bash
cd backend && flask db migrate -m "add deleted_at to simulation_bots" && flask db upgrade
```

### ⚠️ 关键陷阱 2：软删除需同步更新三处查询

添加 `deleted_at` 字段后，必须同步更新以下三处，否则逻辑不一致：
1. **`list_bots`**：过滤 `deleted_at.is_(None)` → Task 3
2. **`create_bot` 槽位计数**：已删除机器人不占槽位 → Task 4
3. **`_run_daily_simulation` Celery 任务**：已删除机器人不参与执行 → Task 7

### ⚠️ 关键陷阱 3：`PATCH` 端点只允许 `"active"` 和 `"paused"` 两个状态

业务上，`"stopped"` 状态是历史遗留设计（`STATUS_LABELS` 里有这个 key），但通过 PATCH 端点不可将机器人设置为 `"stopped"`。`ALLOWED_PATCH_STATUSES = {'active', 'paused'}`，传入其他值应返回 `422`。

### ⚠️ 关键陷阱 4：Celery 定时任务的 `filter_by(status='active')` 已自然跳过 `paused` 机器人

`_run_daily_simulation()` 中 `filter_by(status='active')` 不会选出 `paused` 机器人，行为正确。但添加软删除后仍需增加 `deleted_at.is_(None)` 过滤，防止将来有 `status='active'` 的机器人被意外软删（极端 race condition）。

### ⚠️ 关键陷阱 5：前端 `confirm()` 是同步阻塞

删除确认使用浏览器原生 `confirm()` 对话框，这在项目现有代码中已有先例（见 `3-5` story 的实现）。如果产品要求统一样式，可替换为 Modal 组件，但本 Story 使用 `confirm()` 即可满足需求。
