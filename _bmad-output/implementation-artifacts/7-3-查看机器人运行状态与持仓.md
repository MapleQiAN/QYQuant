# Story 7.3: 查看机器人运行状态与持仓

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为已登录用户，
我希望查看模拟机器人的运行状态和当前持仓详情，
以便了解模拟组合的实时持仓情况。

## Acceptance Criteria

1. **Given** 用户访问模拟托管页面
   **When** 调用 `GET /api/v1/simulation/bots`
   **Then** 返回当前用户所有机器人列表，含 `status`、策略名称、初始资金、创建时间（北京时间，FR43）
   **And** 若用户无任何机器人，返回空数组 `{"data": []}`

2. **Given** 用户点击某个机器人
   **When** 调用 `GET /api/v1/simulation/bots/:id/positions`
   **Then** 返回该机器人当前模拟持仓列表（symbol、持仓数量、平均成本，FR44）
   **And** 若该机器人尚无持仓记录（尚未执行过任何模拟交易），返回空数组 `{"data": []}`

3. **Given** 用户尝试访问不属于自己的机器人持仓
   **When** 调用 `GET /api/v1/simulation/bots/:id/positions`（`:id` 是其他用户的机器人）
   **Then** 返回 `404` 和错误码 `BOT_NOT_FOUND`

4. **Given** 用户未登录
   **When** 访问 `GET /api/v1/simulation/bots` 或 `GET /api/v1/simulation/bots/:id/positions`
   **Then** 返回 `401 Unauthorized`

5. **Given** 用户进入模拟托管页面
   **When** 页面加载完成
   **Then** 前端自动调用 `GET /api/v1/simulation/bots`，展示机器人卡片列表（含 status、策略名称、初始资金、创建时间）

6. **Given** 用户点击某个机器人卡片
   **When** 弹出持仓详情
   **Then** 调用 `GET /api/v1/simulation/bots/:id/positions`，展示持仓列表（symbol、数量、平均成本）

## Dev Notes

### 架构约束与关键模式

- **蓝图文件**：`backend/app/blueprints/simulation.py`（Story 7.1 已创建，本 Story 在同一文件追加端点）
- **认证**：所有端点均需 `@jwt_required()`，`get_jwt_identity()` 获取 `user_id`（字符串）
- **响应格式**：`ok(data)` 封装，data 可为 list 或 dict；错误用 `error_response(code, message, status)`
- **时区**：`created_at` 和 `updated_at` 均用 `format_beijing_iso(datetime_obj)` 转为北京时间 ISO 字符串
- **前端状态**：Pinia store 是唯一数据源，组件不直接调用 API（通过 store actions）
- **TypeScript**：`quantity`、`avg_cost`、`initial_capital` 均为 `string`（后端 Numeric → JSON string）

### ⚠️ 关键陷阱：Strategy.title 字段名须确认

`GET /bots` 需要 join Strategy 获取策略名称。`strategy_name` 字段来源为 `Strategy.title`（架构文档定义）。

**开发前必须**读 `backend/app/models.py` 中 `Strategy` 类，确认字段名是 `title`、`name` 还是其他。不要假设。

```python
# 根据实际字段名选择：
strategy_name = strategy.title   # 可能是 title
# 或
strategy_name = strategy.name    # 或者是 name
```

### ⚠️ 关键陷阱：GET /bots/:id/positions 必须做 authorization check

不能只根据 `bot_id` 查询，**必须同时验证 `user_id`**：
```python
bot = SimulationBot.query.filter_by(id=bot_id, user_id=user_id).first()
if not bot:
    return error_response('BOT_NOT_FOUND', '机器人不存在或无权访问', 404)
```

返回 404 而不是 403，避免暴露资源的存在性（安全实践，不告诉攻击者 bot 是否存在）。

### ⚠️ 关键陷阱：Simulation.ts 类型须向后兼容更新

`SimulationBot` 接口在 Story 7.1 中已定义，缺少 `strategy_name`。本 Story 在接口中追加该字段：
```typescript
strategy_name: string   // ← 新增
```

由于 `createSimBot` API（Story 7.1）返回的响应中可能没有 `strategy_name`（只有 `POST /bots` 创建时），需检查 `createBot` action 中 `this.bots.unshift(bot)` 后 bot 对象是否包含 `strategy_name`。如果不包含，`unshift` 后应立即刷新列表（`await this.fetchBots()`）。

**推荐做法**：在 `createBot` action 成功后调用 `fetchBots()` 刷新，而不是手动 `unshift`：
```typescript
async createBot(payload: CreateBotPayload): Promise<SimulationBot> {
  this.isLoading = true
  try {
    const bot = await createSimBot(payload)
    await this.fetchBots()  // 刷新完整列表（含 strategy_name）
    return bot
  } finally {
    this.isLoading = false
  }
}
```
