## Dev Notes

### 核心设计决策

**1. GoldStepByStep 策略重写 — 从交互式到事件驱动**

现有的 GoldStepByStep 策略存在严重问题：
- 使用 `input()` 获取价格数据（不可回测）
- 自定义 `Date`、`Record_Price`、`Holding`、`Account` 数据类（与 SDK 不兼容）
- `main()` 入口函数（不符合 event_v1 接口）
- 所有策略参数硬编码（不可配置）

重写后的策略应遵循标准 event_v1 模式：

```python
"""黄金阶梯突破策略 — 基于前高突破和回撤止损的趋势跟踪策略。

策略逻辑：
1. 买入条件：当前价格突破指定周期内的历史最高价（前高）
2. 卖出条件（二选一触发）：
   a. 单日跌幅超过阈值（默认 3%）
   b. 从周期新高回撤超过阈值（默认 6%）
"""

from __future__ import annotations

from qysp import BarData, Order, StrategyContext


def on_bar(ctx: StrategyContext, data: BarData) -> list[Order]:
    """每根 K 线触发的回调函数。

    Args:
        ctx: 策略上下文，提供账户信息、参数和下单方法。
        data: 当前 K 线数据（OHLCV + datetime）。

    Returns:
        订单列表。空列表表示不操作。
    """
    # 从 ctx.parameters 读取策略参数（零硬编码）
    drop_one_day_pct = ctx.parameters.get("drop_one_day_pct", 3.0)
    drop_from_peak_pct = ctx.parameters.get("drop_from_peak_pct", 6.0)

    orders: list[Order] = []
    symbol = data.symbol
    price = data.close

    # 检查是否持有该标的
    position = ctx.account.positions.get(symbol)
    has_position = position is not None and position.quantity > 0

    if not has_position:
        # 无持仓 — 检查突破买入条件
        # 注：实际回测中 breakout 高点判断需要历史数据支撑
        # 这里简化为：由平台在策略参数中提供前高价格
        breakout_price = ctx.parameters.get("breakout_price", 0.0)
        if price > breakout_price and breakout_price > 0:
            # 全仓买入
            qty = ctx.account.cash / price
            if qty > 0:
                orders.append(ctx.buy(symbol, qty))
    else:
        # 有持仓 — 检查止损条件
        # 条件 1：单日跌幅超过阈值
        if data.open > 0 and (data.open - price) / data.open * 100 > drop_one_day_pct:
            orders.append(ctx.sell(symbol, position.quantity))

        # 条件 2：从持仓最高价回撤超过阈值
        elif position.avg_cost > 0:
            peak = max(data.high, position.current_price)
            drawdown = (peak - price) / peak * 100
            if drawdown > drop_from_peak_pct:
                orders.append(ctx.sell(symbol, position.quantity))

    return orders
```

**注意**：上述代码为设计参考。实际实现时需要注意：
- 突破前高的判断在真实回测中应基于历史 K 线数据（需要 lookback 窗口）。由于当前 event_v1 接口仅传入当前 K 线（无历史序列），突破逻辑可简化为参数驱动或使用 ctx 提供的历史缓存（如果可用）
- 策略重写的核心目标是**展示 SDK 最佳实践**，而非完美复现原有逻辑
