# Story 9.2: 邮件通知（异步发送）

Status: ready-for-dev

## Story

作为平台，
我希望通过邮件异步发送重要系统通知，
以便用户即使未打开平台也能及时获知关键信息。

## Acceptance Criteria

1. **Given** 策略审核结果产生（approved 或 rejected）
   **When** 管理员操作触发审核完成
   **Then** 系统通过 Celery 任务异步调用 Flask-Mail 发送审核结果邮件（FR60）
   **And** 邮件主题清晰标注平台名称和通知类型（如 `【QYQuant】您的策略审核结果`）
   **And** 邮件发送失败时记录 error 日志，不影响主请求响应（异步容错）
   **And** 同一事件不重复发送邮件（幂等保护）

2. **Given** 邮件服务（SMTP）不可用
   **When** Celery 任务执行邮件发送
   **Then** 最多重试 3 次（指数退避），重试耗尽后记录 error 日志，任务标记为失败

3. **Given** Flask-Mail 已通过 `extensions.py` 和 `__init__.py` 完成初始化
   **When** 应用启动
   **Then** 邮件配置从环境变量读取（`MAIL_SERVER`、`MAIL_PORT`、`MAIL_USE_TLS`、`MAIL_USERNAME`、`MAIL_PASSWORD`、`MAIL_DEFAULT_SENDER`），缺失时应用仍可正常启动（邮件服务降级）

4. **Given** 其他业务模块需要发送邮件通知
   **When** 调用 `send_email_notification.delay(user_id, event_type, context_data)`
   **Then** 任务被推入 Celery 队列异步处理，调用方立即返回，不阻塞主请求

### 前置依赖（Story 9.1）

本 story 依赖 Story 9.1 完成以下基础设施：
- `Notification` 模型（`backend/app/models.py`）
- `create_notification()` 工具函数（`backend/app/utils/notification.py`）

**重要**：Story 9.2 的邮件通知与 Story 9.1 的站内通知是**两个独立渠道**。同一事件（如策略审核）通常需要同时触发两种通知，但各自在不同地方实现（站内通知由 `create_notification()` 插入数据库，邮件通知由 `send_email_notification.delay()` 异步发送）。**本 story 不负责触发逻辑**，仅提供邮件发送基础设施。

Celery beat schedule 只在 `run_daily_simulation` 中，邮件任务是 **on-demand 任务**，不需要加入 beat_schedule。

**Celery task 写法参考**（`backend/app/tasks/simulation_tasks.py`）：
```python
from ..celery_app import celery_app
from ..extensions import db

@celery_app.task(bind=True, max_retries=3)
def send_email_notification(self, user_id, event_type, context_data):
    try:
        # ... 发送逻辑
    except Exception as exc:
        logger.error(f"邮件发送失败 [{event_type}]: {exc}")
        raise self.retry(exc=exc, countdown=60 * (2 ** self.request.retries))
```

### Flask-Mail 初始化

`backend/app/extensions.py` 当前只有 `db, migrate, jwt, cors, api`，**无 Flask-Mail**。需要新增：
```python
from flask_mail import Mail
mail = Mail()
```

然后在 `backend/app/__init__.py` 中加入 `mail.init_app(app)`（检查现有 init 顺序，在 db.init_app 之后）。

**邮件配置降级策略**：若 `MAIL_SERVER` 未配置，Flask-Mail 实例化后 `send()` 调用会抛出异常，但 `init_app` 不会失败 —— 这正是我们希望的行为（应用正常启动，仅邮件发送时报错，由 Celery 任务的 retry 机制处理）。

### 幂等保护实现细节

推荐使用 Redis 的 SET NX 操作（`SETNX email_sent:{event_type}:{target_id} 1 EX 86400`）。

```python
import redis
import os

_redis = redis.from_url(os.getenv('REDIS_URL', 'redis://localhost:6379/0'))

def _is_already_sent(event_type: str, target_id: str) -> bool:
    key = f"email_sent:{event_type}:{target_id}"
    # SET NX: 仅当 key 不存在时返回 True（首次发送）
    return not _redis.set(key, 1, nx=True, ex=86400)
```

注意：若 Redis 不可用（如测试环境），需要有降级处理（try/except，降级时直接发送，不做幂等检查）。

### 关键架构约束

- **异步非阻塞**：邮件发送必须通过 `send_email_notification.delay()` 调用，**禁止**在主请求线程中同步发送邮件
- **无前端改动**：本 story 纯后端，无任何前端变更
- **触发点不在本 story**：本 story 只建立基础设施；具体在策略审核（Epic 10）、支付回调（Story 8.3）等场景调用
- **独立于站内通知**：邮件通知和站内通知是两套独立机制，可同时触发也可单独触发

### Project Structure Notes

- `notification_tasks.py` 放入 `backend/app/tasks/`（与 `backtests.py`、`simulation_tasks.py` 同级）
- 邮件模板放入 `backend/app/utils/email_templates.py`，不使用 Jinja2 文件模板（保持简单，纯 Python 字符串）
- 无新数据库表，无 migration 需求（本 story 无数据库改动）
