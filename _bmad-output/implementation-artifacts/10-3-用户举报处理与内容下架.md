# Story 10.3: 用户举报处理与内容下架

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

作为管理员，
我希望处理用户举报，可以下架违规策略并通知作者，
以便维护平台内容合规性。

## Acceptance Criteria

1. **用户提交举报：** `POST /api/v1/marketplace/strategies/:id/report` 已登录用户可提交举报，请求体包含 `reason`（举报原因，必填，字符串 10~500 字符）；同一用户对同一策略 24 小时内不可重复举报（返回 409）；不可举报自己的策略（返回 400）；举报成功返回 201。
2. **举报数据模型：** 新建 `reports` 表（id、reporter_id、strategy_id、reason、status='pending'|'reviewed'|'dismissed'、admin_note、created_at、reviewed_at、reviewed_by），举报记录写入后 status 默认 pending。
3. **管理员查看举报队列：** `GET /api/v1/admin/reports` 仅 admin 可访问；支持 `status` 筛选（默认 pending）和分页（page/per_page）；返回举报列表含举报者昵称、策略标题、策略作者、举报原因、举报时间，按创建时间倒序。
4. **管理员处理举报 — 下架策略：** `PATCH /api/v1/admin/reports/:id/resolve` 仅 admin 可访问；请求体 `action` 为 `takedown`（下架策略）或 `dismiss`（驳回举报）；`admin_note` 可选；下架时 `strategies.is_public=false` 且发送站内通知给策略作者说明下架原因；驳回时仅更新举报状态；操作写入 `audit_logs`。
5. **策略下架通知：** 下架操作触发站内通知（type="strategy_takedown"）给策略作者，内容包含策略名称和下架原因；同时异步发送邮件通知（复用现有 Celery 邮件任务）。
6. **前端实现：** 管理员后台新增举报管理页面 `/admin/reports`，展示 pending 举报列表；支持下架/驳回操作；下架时可输入备注；操作后局部刷新列表并给出 toast 反馈。用户端策略详情页新增"举报"按钮（仅非作者已登录用户可见），点击弹出举报原因输入框。

**Admin 骨架已存在，直接扩展**
- `backend/app/blueprints/admin.py` 已有策略审核接口（GET 列表 + PATCH 审核），本 Story 新增举报管理接口应遵循完全相同的模式。[Source: backend/app/blueprints/admin.py]
- `@require_admin` 是唯一的 admin 权限门禁，不要新建第二套鉴权。[Source: backend/app/utils/auth_helpers.py]

**通知与审计能力优先复用，不要重新实现**
- 站内通知：`backend/app/services/notifications.py:create_notification(user_id, type, title, content)` [Source: backend/app/services/notifications.py]
- 异步邮件：`backend/app/tasks/notification_tasks.py:send_email_notification.delay(user_id, event_type, context_data)` [Source: backend/app/tasks/notification_tasks.py]
- 审计日志：`backend/app/utils/audit.py:log_audit(operator_id, action, target_type, target_id, details)` [Source: backend/app/utils/audit.py]
- 邮件模板：`backend/app/utils/email_templates.py`，参照 `render_strategy_review_email()` 实现新的 `render_strategy_takedown_email()` [Source: backend/app/utils/email_templates.py]

**策略模型字段已存在**
- `Strategy.is_public` 和 `Strategy.review_status` 已有，下架操作只需设置 `is_public=False`，不需要改 `review_status`。[Source: backend/app/models.py]
- marketplace 可见性条件 `is_public=True AND review_status='approved'`，下架后自动从广场消失。[Source: backend/app/blueprints/marketplace.py]

**前端参照 StrategyReview.vue 模式**
- `frontend/src/views/admin/StrategyReview.vue` 已建立 admin 列表页 + 操作按钮 + toast 反馈的完整模式，举报管理页应完全参照。[Source: frontend/src/views/admin/StrategyReview.vue]

**邮件发送幂等性**
- `send_email_notification` 使用 `email_sent:{event_type}:{target_id}` Redis key 做 24h 去重。对于下架通知，`target_id` 应使用 strategy_id 确保同一策略多次举报不会重复发邮件。[Source: backend/app/tasks/notification_tasks.py]

### 上一个 Story（10.2 策略审核队列）学到的关键模式

- 审核接口参数校验：status 白名单 + reason 条件必填 + 去首尾空格
- 事务边界：策略状态变更、通知、审计日志在同一 `db.session.commit()` 中提交
- 前端乐观更新：操作成功后立即从列表中移除已处理项，无需重新请求
- 分页元数据：`{ total, page, per_page }` 标准格式
- 时间字段：使用 `format_beijing_iso_ms()` 格式化毫秒时间戳，`now_utc()` 用于 DateTime 字段

### 实现边界与注意事项

- **新增 Report 模型**是本 Story 的唯一新表，不要在 Strategy 模型上新增举报相关字段。
- 用户举报接口放在 `marketplace.py` Blueprint（与策略广场功能同域），管理员处理接口放在 `admin.py`。
- 下架操作只修改 `is_public=False`，不改变 `review_status`（下架 ≠ 审核拒绝）。
- 同一策略被多人举报时，下架该策略后应批量将所有 pending 举报标记为 reviewed，避免管理员重复处理。
- 驳回举报不通知举报者也不通知策略作者，仅记录审计日志。
- 举报接口不暴露策略代码、code_encrypted、code_hash 等敏感信息。
- 用户端"举报"按钮仅对已登录且非策略作者的用户可见。
